# Дублирование сайта под город

**Issue:** [#65](https://github.com/alshchetinin/zhk-starter/issues/65)
**Дата:** 2026-06-14
**Статус:** дизайн

## Проблема

Система мультитенантная: к сайту (`sites`, есть `cityId`) привязано по `site_id` почти всё —
контент (pages, homepage, modals, news, promotions, documents, contacts, social, purchase-methods,
mortgage, banks, page-categories, ticket-settings, SEO) и каталог недвижимости
(projects/buildings/apartments/...). Каждый город-сайт сейчас заполняется **с нуля** — долго.

## Цель

Кнопка **«Дублировать сайт»** создаёт новый сайт под город и копирует **контент-шаблон**, чтобы
не строить структуру/контент заново. Каталог недвижимости у города свой (приходит своим синком),
поэтому не копируется.

## Не-цели (вне скоупа)

- Не копируем каталог недвижимости и интеграции (город синкает свой каталог отдельно).
- Не делаем наследование/мастер-шаблон (правки исходного сайта НЕ распространяются в копии —
  копия независима). Это сознательный выбор подхода A (клон-однократно) из брейншторма; B/C
  (наследование, экспорт/импорт) — отдельно при необходимости.
- Не копируем историю версий контента (`content-versions`) — только текущий контент.

## Подход: клон-однократно (A)

Одна транзакционная операция: создать новый сайт → скопировать контент-таблицы → построить карту
`старый id → новый id` → переписать FK и ссылки внутри JSONB-блоков на новые id. Копия живёт
независимо.

## Что копируется / что нет

**Копируется** (контент/шаблон, с ремапом FK на новые id):

| Таблица | Примечание |
| --- | --- |
| `sites` | новая строка (см. «Поля нового сайта») |
| `contacts` | город-специфичны (телефон/адрес) — копия как стартовый шаблон, потом правишь |
| `page_categories` | → `pages.categoryId` ремапится |
| `banks` | конфиг ипотечного калькулятора |
| `mortgage_programs` | копируется после banks; `bankId` ремапится. Join `mortgage_program_projects` (→ projects) НЕ копируется |
| `purchase_methods` | join `purchase_method_projects` (→ projects) НЕ копируется |
| `social_links` | |
| `ticket_settings` | |
| `documents` | город-специфичны — копия как шаблон |
| `pages` | `categoryId` ремап + блоки (см. ниже) |
| `homepage` | одна строка per-site; блоки ремап |
| `modals` | блоки ремап (если есть `contentBlocks`) |
| `news` | блоки ремап |
| `promotions` | блоки ремап |
| `site.settings` (JSONB) | SEO/analytics копируются; `contactsHeaderIds`/`contactsFooterIds` ремап на новые контакты; `analytics.yandexMetrika.counterId` очищается |

**НЕ копируется**: весь каталог недвижимости (`projects`, `buildings`, `sections`, `entrances`,
`floors`, `non_residential_floors`, `apartments`, `apartment_layouts`, `commerce`, `parking`,
`storage`, `construction_progress`, `decorations`, `apartment_tags`, `apartment_layout_tags`),
`tags` (приходят с синком каталога), `integrations`, `content_versions`.

> **Решения на ревью (можно поменять):** `news`/`promotions`/`documents` — город-специфичный
> контент; по умолчанию КОПИРУЕМ (трактуем «вместе с контентом» буквально, лишнее удалишь под
> город). Если хочешь — исключим из копии (новый город пишет своё). `tags` по умолчанию НЕ
> копируем (катало-смежные, приходят с синком) — если нужны как шаблон фильтров, включим.

## Ремап ссылок в блоках (ядро сложности)

Контент-блоки хранятся как JSONB (`contentBlocks`). У блоков есть relation-поля двух типов
(из `defineBlock` fields):

- **`project`** (поля типа `project`, блоки `category: "project"`) — хранят `projectId`.
  Каталог НЕ копируется → **оставляем id как есть**. Резолвится кросс-сайтово по id
  (`enrichContentBlocks` фильтрует по `projects.id`, не по siteId — проверено), покажет исходный
  проект, пока не перелинкуешь под город.
- **`contacts`** (поля типа `contacts`) — хранят id контакта(ов) → **ремап** на новые
  скопированные контакты (по карте старый→новый contactId).

Хелпер `remapBlockReferences(blocks, { contacts: Map<oldId,newId> }, blockFieldIndex)`:
для каждого блока берёт определение его типа, проходит поля; `contacts`-поля переписывает по
карте, `project`-поля и остальное (image/strings/...) оставляет. Чистая функция —
покрывается unit-тестами.

## Поля нового сайта

- `id` — новый UUID; `name`/`slug`/`cityId` — из формы (slug уникален, валидация);
- `isPrimary = false`, `isActive = false` (черновик — активируешь, когда заполнишь город-специфику);
- `customDomain = null`, `accessPassword = null`;
- `settings` — копия исходных с ремапом контакт-id и очищенным Metrika `counterId`.

## Порядок копирования (FK-зависимости)

1. `sites` (новая строка).
2. Без зависимостей внутри контента: `contacts`, `banks`, `page_categories`, `social_links`,
   `purchase_methods`, `ticket_settings`, `documents`.
3. `mortgage_programs` (после `banks`; ремап `bankId`).
4. С блоками/FK: `pages` (ремап `categoryId` + блоки), `homepage`, `modals`, `news`,
   `promotions` (ремап блоков).
5. Собрать `site.settings` с ремапленными контакт-id, записать в новый `sites`.

Каждый шаг строит запись в карте `oldId → newId` своей сущности.

## Архитектура (изоляция)

- **`packages/api/src/services/site-duplication.ts`** — сервис дублирования: оркестратор
  `duplicateSite(tx, { sourceSiteId, name, slug, cityId })` + маленькие per-entity копир-функции +
  чистый `remapBlockReferences`. Источник истины по тому, что копируется.
- **oRPC процедура** `sites.duplicate` (`adminProcedure`) — валидирует вход (slug уникален, sourceSite
  существует), вызывает сервис в `db.transaction`, возвращает новый сайт.
- **Admin UI** — действие «Дублировать» в списке/детали сайта ([sites/index.vue](../../../apps/admin/app/pages/sites/index.vue),
  [sites/[id]](../../../apps/admin/app/pages/sites/[id])) → `UModal` (имя, slug авто из имени,
  город из `cities`), сабмит → мутация → тост + переход на новый сайт. Кнопка/раздел — только
  admin (создание сайтов).

## Выполнение: синхронно в транзакции

Каталог (большие таблицы: apartments и т.п.) исключён, поэтому объём копий умеренный
(контент-слой — десятки/сотни строк). Делаем **синхронно** в `db.transaction` с loading-стейтом и
блокировкой кнопки. Если контента в проде окажется очень много (тысячи news) и операция станет
заметно медленной — вынесем в фоновую задачу отдельно (YAGNI сейчас).

## Edge cases

- **slug-уникальность** — валидируем до вставки (есть unique-констрейнт; отдаём понятную ошибку).
- **Транзакционность** — весь клон в одной транзакции: при ошибке откат, полусайтов не остаётся.
- **Пустой исходный сайт** — копия пустая, без ошибок.
- **Блок ссылается на удалённый проект** — id остаётся, `enrichContentBlocks` отфильтрует
  (как и сейчас на исходном сайте) — не наша забота.
- **`integrationId`** на копируемых строках — у контента он null (контент ручной), у каталога не
  копируем; конфликтов с синком нет.

## Тестирование

- **Unit (детерминированное, vitest в репо):** `remapBlockReferences` (contacts ремапятся,
  project остаётся, неизвестные поля не трогаются); построение карты id; что `bankId`/`categoryId`
  ремапятся.
- **Ручная проверка:** дублировать сайт → у нового сайта есть скопированные страницы/главная/блоки;
  контакт-ссылки указывают на новые контакты; project-ссылки указывают на исходные проекты
  (кросс-сайт, до перелинка); каталог пуст; `settings.contacts*Ids` ремапнуты, `counterId` пуст;
  `isActive=false`; slug-конфликт даёт ошибку; откат при сбое.
