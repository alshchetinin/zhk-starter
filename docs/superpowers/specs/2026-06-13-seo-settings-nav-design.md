# Вынос SEO в отдельный раздел + пункты SEO/Настройки в подменю сайта

- **Дата:** 2026-06-13
- **Issue:** [#61](https://github.com/alshchetinin/zhk-starter/issues/61)
- **Ветка:** `feat/61-seo-settings-nav`
- **Контекст:** продолжение SEO-слоя ([#60](https://github.com/alshchetinin/zhk-starter/issues/60))

## Проблема

SEO-настройки сайта живут одной карточкой внутри общей страницы настроек
`apps/admin/app/pages/sites/[id].vue` вместе с Основными/Доступом/Контактами/
Метрикой. Страница растёт, а навигироваться к SEO неудобно — нет прямого пункта
в меню. Нужно вынести SEO в самостоятельный раздел и дать прямые ссылки на
SEO и Настройки в подменю сайта, чтобы упростить навигацию и заложить
расширяемость (в будущем — анализ всех страниц внутри SEO).

## Решение

### 1. Маршруты и файловая структура

Текущий `apps/admin/app/pages/sites/[id].vue` разбивается на каталог
`apps/admin/app/pages/sites/[id]/`:

- **`settings.vue`** — текущая страница настроек **минус** SEO-карточка
  (Основные, Доступ, Контакты на сайте, Яндекс.Метрика, события Метрики,
  SocialLinksEditor).
- **`seo.vue`** — извлечённая SEO-карточка как самостоятельная страница.
  Несёт **исправленный** селект «Контакт для телефона и адреса»: без
  sentinel-пункта `value: ""` (он нарушает инвариант Reka UI —
  `ComboboxItem`/`SelectItem` бросают ошибку на пустую строку), с пропом
  `clear` для сброса к «Авто». Источник правки — ветка
  `fix/60-seo-org-contact-select`, которая после этого становится не нужна.
- **`index.vue`** — редирект на `./settings`, чтобы существующие ссылки
  `/sites/{id}` (из списка «Сайты» — `sites/index.vue`) продолжали работать.

Решение по маршрутизации: **per-id роуты с прямым SPA-переходом** (а не
глобально-скоупленные `/seo`/`/settings` по текущему сайту). Клик по пункту
открывает страницу конкретного сайта по URL `:id`, без переключения активного
сайта и без полной перезагрузки (в отличие от контент-пунктов, которые ходят
через `setSite` с `window.location`).

### 2. Данные и сохранение

Обе страницы:
- грузят сайт через `sites.getById` по `route.params.id`;
- сохраняют через тот же `sites.update`.

`sites.update` мержит `settings` поверхностно (`{ ...existing, ...settings }`),
поэтому каждая страница шлёт только свой срез без затирания чужого:
- `settings.vue` → top-level (`slug`, `name`, `customDomain`, `isActive`,
  `accessPassword`) + `settings: { contactsHeaderIds, contactsFooterIds,
  analytics }`;
- `seo.vue` → `settings: { seo }`.

Каждая страница держит свой `form`/`watchEffect`/`updateMutation` со своим
срезом полей. Логика и тосты — как в текущей странице.

### 3. Подменю сайта (`AppSidebar.vue`)

В дереве сайтов, в блоке `<nav>` под `visibleContentItems` каждого
развёрнутого сайта, после контент-пунктов добавляется тонкий разделитель и два
пункта (видны только админам — `v-if="isAdmin"`):

- **SEO** — иконка `i-solar-magnifer-zoom-in-linear`, `NuxtLink` на
  `/sites/{site.id}/seo`;
- **Настройки** — иконка `i-solar-settings-linear`, `NuxtLink` на
  `/sites/{site.id}/settings`.

Это обычные `NuxtLink` (прямой переход), а не кнопки `openSiteContent` —
переключение активного сайта не происходит. Подсветка активного пункта — по
`isActive(/sites/{site.id}/seo)` и т.п. Видимость только админам соответствует
тому, что весь раздел «Сайты» — `adminOnly`, а `sites.update` —
`adminProcedure` (не-админ всё равно не сохранит).

### 4. Что не трогаем

- `useNavigation.ts` (`contentItems`) — без изменений: контент-пункты остаются
  глобально-скоупленными, новые пункты живут per-id прямо в шаблоне сайдбара.
- Список «Сайты» (`sites/index.vue`), web-приложение, API, БД — без изменений.
- SEO-вывод на публичном сайте уже работает (из #60).

## Расширяемость

SEO теперь отдельный роут `/sites/[id]/seo` — в будущем туда добавляются вкладки
или секции (анализ всех страниц сайта, проверка мета-тегов и т.п.) без влияния
на страницу настроек.

## Вне объёма

- Вкладочный интерфейс внутри SEO/Настройки — не делаем (отдельные пункты меню).
- Глобально-скоупленные роуты `/seo`/`/settings` — отвергнуто в пользу per-id.
- Изменение прав доступа/новые procedure — не делаем (остаётся admin-only через
  существующий `adminProcedure`).
- Реальный «анализ страниц» в SEO — будущая итерация, здесь только каркас
  раздела.
