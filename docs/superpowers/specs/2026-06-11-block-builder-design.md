# Редактор блоков: превью в пикере + dev-редактор схемы полей

**Дата:** 2026-06-11
**Issue:** [#57](https://github.com/alshchetinin/zhk-starter/issues/57)
**Статус:** одобрено пользователем

## Цель

Две фичи для блочной системы:

1. **Превью блока в пикере** — при добавлении блока пользователь видит PNG-скриншот
   (из Figma или скриншот готовой секции), а не только иконку с описанием.
2. **Dev-редактор схемы блоков** — разработчик редактирует поля блока прямо в админке
   (как Strapi content-type builder): добавил поле → оно сразу появилось в форме
   редактирования блока и в TS-типах/Zod-схеме. Доступно только в dev-режиме,
   изменения пишутся в файлы проекта, Vite HMR подхватывает.

Не входит в объём (осознанно):

- Миграция данных контента при изменении схемы (как в Strapi — данные не трогаем).
- Автогенерация вёрстки в web-рендерере для новых полей существующих блоков —
  поле приходит в props, вёрстку разработчик добавляет руками.
- Редактирование схемы в проде.

## Архитектура

### 1. Модель данных: декларативные `fields`

Сейчас определение блока — рукописный Zod, который машина не может надёжно
прочитать и переписать (round-trip проблема). Решение: поля описываются
декларативными данными, а Zod из них эмитится генератором.

В `packages/api/src/shared/blocks/_core.ts` добавляется:

```ts
export type BlockFieldType =
  | "string" | "text" | "richtext" | "number" | "boolean"
  | "url" | "image" | "images" | "select" | "repeater";

export interface BlockField {
  name: string;            // camelCase идентификатор
  type: BlockFieldType;
  label: string;
  required: boolean;
  description?: string;
  options?: string[];      // только для select
  minItems?: number;       // только для repeater
  maxItems?: number;       // только для repeater
  subFields?: BlockField[]; // только для repeater
}
```

- `DefineBlockInput`/`BlockDefinition` получают обязательное `fields: BlockField[]`.
- `BlockField` заменяет дублирующий тип `Field` в `scripts/generate-block/` —
  скрипты импортируют его из `@zhk/api`.
- Файл определения блока по-прежнему содержит **явные** `dataSchema` (Zod) и
  `defaultData` — точная типизация через `z.infer` сохраняется. Но теперь и
  `fields`, и Zod эмитятся генератором из одного источника (BlockInfo), поэтому
  рассинхрон невозможен.
- Определение блока объявляется **генерируемым артефактом**: править руками
  можно, но сохранение из UI перезапишет файл целиком.
- 12 существующих определений единоразово переводятся на новый формат —
  по фактическим `dataSchema` в TS (не по устаревшим JSON-конфигам в
  `design/blocks/`).

### 2. Кодген: остаётся в `scripts/generate-block/`

> Поправка после ревизии кода: dev-инфраструктура частично уже реализована
> (oRPC-роутер `dev.blocks`, страницы `/dev/blocks`), и она уже импортирует
> генераторы из `scripts/generate-block/` напрямую. Отдельный пакет
> `packages/block-codegen` не нужен — следуем существующему паттерну.

Изменения в генераторах:

- `block-definition.ts` дополнительно эмитит `fields` (декларативные данные)
  и получает **update-режим**: перезапись существующего определения без
  повторной регистрации в `index.ts`;
- `generateEditorComponent` уже перезаписывает файл безусловно — переиспользуется
  для update как есть;
- тип `FieldInfo` в скриптах становится алиасом `BlockField` из `@zhk/api`
  (type-only импорт) — один источник правды для формы поля.

CLI `pnpm generate:block` не меняет поведение и интерфейс.

### 3. Dev-API: oRPC-роутер `dev.blocks` (существующий + новые процедуры)

Роутер `packages/api/src/routers/dev/blocks.ts`, все процедуры на
`devProcedure`: guard `NODE_ENV !== "production"` + авторизация + роль admin
(строже, чем планировалось изначально).

| Процедура | Статус | Действие |
| --- | --- | --- |
| `dev.blocks.list` | есть | Метаданные блоков с диска (regex-parse TS). |
| `dev.blocks.create` | есть | Создать блок: 3 файла + регистрация в `index.ts`. |
| `dev.blocks.update` | **новая** | Переписать определение (с `fields`) + перегенерировать admin-редактор. Web-рендерер **не трогается**. Тип блока неизменяем (v1). |
| `dev.blocks.delete` | есть, дополняется | Удалить 3 файла + дерегистрация + **удалить превью-PNG**. |
| `dev.blocks.uploadPreview` | **новая** | base64 PNG (с проверкой magic bytes) → `apps/admin/public/block-previews/{type}.png`. |

Отдельный GET полей не нужен: admin импортирует `allBlocks` из `@zhk/api`
(включая `fields` после миграции), после записи файлов Vite HMR пересобирает
пакет — клиент видит свежие `fields` автоматически (~1–2 сек). Это и есть
«сразу подтянулось».

### 4. UI: раздел «Блоки» (`/dev/blocks`, только dev)

Раздел «Разработка» в навигации уже существует и показывается при
`import.meta.dev` (`useNavigation.ts`).

**`/dev/blocks`** — список блоков (есть, дополняется): ссылка на страницу
редактирования, мини-превью. Кнопка «Создать блок» (есть).

**`/dev/blocks/create`** — создание блока (есть). Форма выносится в общий
компонент `BlockSchemaForm` (мета + список полей + subFields repeater +
перестановка порядка), переиспользуемый страницей редактирования.

**`/dev/blocks/{type}`** — **новая** страница редактирования, две секции:

- *Мета*: label, description, icon (UInput + живой предпросмотр иконки),
  category (USelect), превью — текущая картинка + загрузка PNG.
- *Поля*: список полей в порядке отображения в форме. Для каждого: имя
  (camelCase, валидация), тип, label, required, description, options (select),
  minItems/maxItems и вложенные subFields (repeater). Добавление, удаление,
  редактирование, перестановка порядка.

Поведение:

- «Сохранить» → `dev.blocks.update` → toast «Схема обновлена».
- Переименование поля — inline-предупреждение: данные в сохранённых страницах
  останутся под старым ключом.
- Тип блока (`type`) на странице редактирования неизменяем (v1) — переименование
  типа потребовало бы переименования файлов и миграции контента в БД.
- Новый блок — та же форма + поле `type` (kebab-case, проверка уникальности)
  → `dev.blocks.create` (есть).
- Удаление — confirm с предупреждением, что контент этого типа в страницах
  перестанет рендериться → `dev.blocks.delete` (есть).

Всё на компонентах @nuxt/ui по правилам проекта (CLAUDE.md).

### 5. Пикер с превью

В `BlockPicker.vue` элементы списка становятся карточками: сверху
`<img src="/block-previews/{type}.png">`, под ней label + description.

- Наличие картинки — по конвенции имени файла, **без поля в определении**:
  `@error` на `<img>` прячет картинку и возвращает текущий вид с иконкой.
  Ноль связности, не нужно переписывать определение при загрузке превью.
- Slideover расширяется, чтобы превью читались.

### 6. Совместимость контента в БД (JSONB)

Данные не мигрируются, но система им терпима:

- **Новое поле**: при загрузке контента страницы в admin-редактор каждый блок
  нормализуется `{ ...defaultData, ...data }` (одно место — где контент входит
  в форму). На web — тот же merge в зоне рендера блоков; рендереры и так терпят
  `undefined` для optional-полей.
- **Удалённое поле**: лишний ключ в JSONB отбрасывается `z.object` при parse.
- **Смена типа поля**: данные не конвертируются — разработчик проверяет контент
  руками (v1, осознанно, как в Strapi).

### 7. Ошибки

- BlockInfo валидируется на сервере Zod-схемой: имена — валидные идентификаторы,
  `select` требует `options`, `repeater` требует `subFields`, `type` блока —
  kebab-case.
- Конфликт `type` при создании → 409.
- Ошибка записи файлов → 500 с текстом в toast; файлы пишутся последовательно,
  сообщение указывает упавший шаг.

## Тестирование

В репозитории пока нет тестовой инфраструктуры — добавляется vitest на уровне
корня (devDependency + `vitest.config.ts` + скрипт `pnpm test`).

- Unit-тесты генераторов (`scripts/generate-block/__tests__/`): snapshot
  сгенерированных файлов (определение, редактор, рендерер), регистрация в
  index.ts, update-режим (перезапись без дублей в index.ts).
- Unit (`packages/api`): consistency-тест «имена `fields` каждого блока ==
  ключи `dataSchema` и `defaultData`» — механически ловит дрейф при миграции
  12 определений; тест normalize-merge.
- CLI-генератор и dev-роутер используют те же функции — покрываются теми же
  тестами.
- Ручная проверка dev-флоу: добавить поле → форма обновилась через HMR;
  создать блок; удалить блок; загрузить превью; пикер показывает картинки
  и fallback-иконки.
