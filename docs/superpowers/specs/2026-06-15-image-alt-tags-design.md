# Alt-теги для изображений: центральный baseline + per-usage override

**Дата:** 2026-06-15
**Issue:** [#66](https://github.com/alshchetinin/zhk-starter/issues/66)
**Статус:** Дизайн утверждён, готов к плану

## Задача

Дать редактору возможность задавать `alt`-текст для **всех** картинок и выводить
его в `<AppImage>` на сайте. Сейчас `alt` нигде не редактируется: в данных блоков
картинка — «голая» URL-строка, а рендереры подставляют в `:alt` соседние поля
(`title`, `name`) или `""`. При этом в БД **уже есть** колонка `media.alt`
([`packages/db/src/schema/media.ts:13`](../../../packages/db/src/schema/media.ts)),
которая никогда не заполнялась и не читалась.

## Принятые решения (итоги брейншторма)

1. **Где живёт alt** — гибрид: центральный baseline на файл **плюс** опциональный
   per-usage override.
2. **Кто задаёт per-usage** — редактор прямо в блоке (override хранится в данных
   блока).
3. **Охват** — блоки **и** все общие загрузчики (`ImageUpload`/`GalleryUpload`):
   галерея проекта, аватары контактов, обложки новостей/акций, теги, банки,
   модалки, og и т.д.
4. **Фазы** — реализуем обе сразу (центральный alt + per-usage в блоках).
5. **Пустой alt** — ненавязчивая подсказка под инпутом, без блокировки сохранения
   (декоративная картинка → пустой alt корректен).

## Архитектура

### Модель хранения (две точки, явный приоритет)

**Центральный `media.alt`** — основной, универсальный механизм. Один alt на
физический файл (по `url`). Покрывает **все ~20 потребителей** общих загрузчиков
без изменения их колонок/схем: они хранят URL, а alt резолвится по URL → `media`.

**Per-usage override** — необязательный доп. слой, **только для image-полей
блоков** (типы поля `image`/`images`, в т.ч. внутри `repeater`) и для
`GalleryItem` (там метаданные уже поддержаны). Значение image-поля расширяется до
union `string | { url: string; alt?: string }` и нормализуется в объект на чтение.

**Приоритет alt на рендере:**

```
per-usage (объект в данных блока)
  → central (media.alt по url)
    → alt, переданный рендерером (title/name — текущее поведение)
      → "" (декоративная)
```

### Редактор (в общих загрузчиках)

- `ImageUpload` и `GalleryUpload` получают **alt-инпут под превью** + ненавязчивую
  подсказку, когда alt пуст (нейтральный текст про SEO/доступность, без ошибки).
- В **строковых** потребителях (аватары, обложки, og и пр.) у инпута нет
  локального слота — он пишет **центральный** `media.alt` через новую ручку
  `media.update({ url, alt })` (дебаунс). Слот per-usage не нужен и не создаётся.
- В **блочных** потребителях инпут пишет **per-usage** в данные блока (объект
  `{ url, alt }`); как placeholder показывается центральное значение, рядом — ссылка
  «Сделать значением по умолчанию для файла» (пушит текущий текст в `media.alt`).
- Один инпут; поведение определяется тем, объектный ли `model` (т.е. умеет ли
  потребитель хранить per-usage). Никакого второго поля и визуального шума.

### Вывод на сайте (`AppImage`)

- `AppImage` делаем **union-aware**: проп `src: string | { url: string; alt?: string }`.
  Внутри извлекаем `url`/`alt`. Тогда рендереры с `:src="image"` **не меняются**, а
  их текущий `:alt="title"` автоматически становится последним фолбэком.
- Центральные alt доставляются на web через **карту URL→alt**: новая ручка
  `media.altMap` (отдаёт только непустые alt; карта глобальная — `media` не
  привязана к сайту), композабл `useImageAlt(url)` читает её. Карта загружается
  один раз и кешируется (SSR-state).
- Итоговая логика в `AppImage`:
  `alt = объектный alt ?? central(url) ?? altProp ?? ""`.
  Карта **обязательна** для строковых потребителей (аватары/обложки) — у них alt
  берётся только из центра.

## Объём работ

### Фаза 1 — центральный alt + вывод (покрывает 100% картинок)

API / DB:
- `media.update({ url, alt })` — `protectedProcedure`, `UPDATE media SET alt WHERE url = ?`
  (url де-факто уникален: `key` содержит uuid). Если строки нет — no-op (картинка
  загружена до трекинга или внешняя).
- `media.altMap()` — `protected/publicProcedure`, возвращает `Record<url, alt>`
  только для непустых alt.

Web:
- `AppImage.vue` — union-aware `src`, резолв alt по цепочке, интеграция
  `useImageAlt`.
- `composables/useImageAlt.ts` + плагин/`useState`-загрузка карты `media.altMap`
  (одна загрузка, кеш). Уточнить точку инъекции в bootstrap web рядом с
  существующими site-композаблами.
- Аудит рендереров на места, где «сырое» значение используется как строка
  (`:key="src"`, вычисления над URL) — там завести `imgUrl(value)` хелпер. Передача
  в `<AppImage :src>` менять не нужно.

Admin:
- `ImageUpload.vue` / `GalleryUpload.vue` — alt-инпут + подсказка; запись
  центрального alt через `media.update` для строковых потребителей.

### Фаза 2 — per-usage override в блоках

Схема/типы:
- Общий Zod-хелпер значения картинки в `packages/api/src/shared/blocks/_core.ts`:
  `imageValue = z.union([z.string().url(), z.object({ url: z.string().url(), alt: z.string().optional() })])`
  и `imagesValue = z.array(imageValue)`. defaultData (`null`/`[]`) остаётся валиден.
- Генератор `scripts/generate-block/field-types.ts` — `image`/`images` эмитят
  ссылку на общий хелпер вместо `z.string().url()`; `tsType` — union; vueTemplate
  без изменений (тот же `ImageUpload`/`GalleryUpload`).
- `scripts/generate-block/generators/{block-definition,web-renderer}.ts` —
  согласовать эмиссию (рендер-stub: `:src` остаётся, т.к. AppImage union-aware).
- `GalleryItem` (`apps/admin/app/types/gallery.ts`) — добавить `alt?: string | null`;
  `GalleryUpload` уже хранит объекты — расширить редактирование на alt.

Round-trip / консистентность:
- Пересобрать канонические определения блоков (`pnpm generate:block` идемпотентно)
  → обновятся `packages/api/src/shared/blocks/*.ts`.
- Обновить тесты `packages/api/src/shared/blocks/__tests__/blocks.test.ts`
  (fields ↔ dataSchema ↔ defaultData) и
  `scripts/generate-block/__tests__/generators.test.ts` (round-trip эмиссии) под
  новый zodType image-полей.
- `normalizeBlockData` — поверхностный мерж не нормализует элементы repeater;
  per-usage alt внутри repeater резолвится на уровне union (чтение), миграция
  данных не требуется.

Editor/renderer:
- Union-aware рендереры: значения image-полей могут быть объектом — `AppImage`
  это покрывает; точечно поправить места строковых операций над значением.

## Тестирование

- **Unit:** `media.update`/`media.altMap` (включая no-op при отсутствии строки,
  фильтр пустых в altMap). Резолв alt-цепочки (чистая функция приоритета).
- **Blocks:** обновлённые consistency/round-trip тесты зелёные после регенерации.
- **E2E (preview, опора на существующую сессию):** задать alt в блоке (per-usage),
  задать alt у аватара/обложки (central), проверить вывод в DOM `<img alt>` на web
  для обоих путей и фолбэк-цепочку.

## Non-goals

- Отдельная страница/раздел «Медиа» для массового редактирования alt (Q3 —
  не выбрано).
- `og:image:alt` и прочие SEO-meta для alt (отдельный SEO-слой).
- Миграция существующих строковых колонок (аватары/обложки) в объекты — они
  остаются строками, alt у них только центральный.
- Подпись/`title`/`figcaption` — вне задачи (это другой контент).

## Открытые мелочи (решить в плане)

- Точка инъекции загрузки `altMap` в web bootstrap и стратегия кеша/инвалидации.
- Нужен ли upsert в `media.update` для внешних/старых URL без media-строки
  (по умолчанию — no-op).
- Дебаунс/он-blur запись центрального alt из `ImageUpload`.
